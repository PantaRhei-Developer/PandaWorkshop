# 命名規則
- 本番環境は「prod」ではなく「prd」としてください。「dev」と文字数を合わせるためです。
- Google Cloudのプロジェクトの命名規則は「組織名-フォルダ名-フォルダ名(-フォルダ名, 省略化)-環境名 (例: pantarhei-oarth-app-prd)

# クレデンシャル管理
- クレデンシャルファイル（APIキー、シークレット、サービスアカウントキー等）は、用途に応じて以下のように管理してください。
  - **ローカル開発時**: 環境変数で管理する（`.env` ファイルを利用し、`.gitignore` に必ず追加してリポジトリへコミットしないこと）
  - **デプロイ後（本番・ステージング等のクラウド環境）**: Google Cloud Secret Manager で管理する
- クレデンシャルをソースコード・設定ファイル・ログ等に直接記載することは禁止します。
- `.env.example` など、変数名のみを記載したテンプレートファイルはリポジトリに含めて構いません。

# データ管理規約
- **データ分類**: 取り扱うデータは以下の区分で分類し、区分ごとに保管・アクセス制御の方針を定めてください。
  - 公開データ / 社内データ / 機密データ / 個人情報（PII）
- **保管場所**:
  - 構造化データは原則 Cloud SQL / BigQuery を利用
  - 非構造化データ（画像・ファイル等）は Cloud Storage を利用
  - 環境（dev/stg/prd）ごとにバケット・データセットを分離すること
- **アクセス制御**:
  - IAM は最小権限の原則（Principle of Least Privilege）に従い、ロールベースで付与
  - 個人アカウントへの直接付与ではなく、Google Group 経由での付与を推奨
  - 本番環境のデータへのアクセスは監査ログを必ず有効化すること
- **個人情報（PII）の取扱い**:
  - 必要最小限の収集に留め、保存時は暗号化（CMEK 推奨）を行うこと
  - 開発・検証環境では本番データをそのまま使用せず、マスキング・匿名化したデータを利用すること
- **バックアップとリテンション**:
  - 重要データは自動バックアップを設定し、リストア手順をドキュメント化すること
  - 法令・社内規程に基づき保持期間を定め、期限を過ぎたデータは削除すること
- **データの受け渡し**:
  - 外部とのデータ授受は、認可された経路（署名付きURL、Secret Manager、暗号化通信等）を経由すること
  - Slack やメール等への直接添付は禁止します。
